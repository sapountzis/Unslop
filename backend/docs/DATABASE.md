## Overview

Unslop uses **Drizzle ORM v1 Beta** with a **code-first approach**. Your TypeScript schema is the source of truth, and Drizzle generates SQL migrations from it.

### Key Concepts

- **Schema**: TypeScript files defining your database structure (`backend/src/db/schema.ts`)
- **Migrations**: SQL files generated by Drizzle Kit (`backend/drizzle/`)
- **Migration Tracking**: Drizzle tracks applied migrations in `drizzle.__drizzle_migrations` table
- **Two Workflows**: `push` for rapid development, `generate + migrate` for production

## Current Setup

### Versions

```json
{
  "drizzle-orm": "1.0.0-beta.13",
  "drizzle-kit": "1.0.0-beta.13"
}
```

### Database Connection

We support **two database drivers**:

1. **Local Development**: `postgres-js` (standard PostgreSQL connection)
2. **Production (Neon)**: `neon-http` (serverless HTTP driver)

**Connection code** (`backend/src/db/index.ts`):

```typescript
import { drizzle as drizzleNeon } from 'drizzle-orm/neon-http';
import { drizzle as drizzlePostgres } from 'drizzle-orm/postgres-js';

const DATABASE_URL = process.env.DATABASE_URL!;

// Auto-detect based on URL
const isNeonDatabase = DATABASE_URL.includes('neon.tech') || DATABASE_URL.includes('neon.com');

function createDb() {
  if (isNeonDatabase) {
    return drizzleNeon(DATABASE_URL);
  } else {
    return drizzlePostgres(DATABASE_URL);
  }
}

export const db = createDb();
```

**Note**: In Drizzle v1 beta, you pass the connection string directly to `drizzle()`, not a client instance.

### Configuration

**File**: `backend/drizzle.config.ts`

```typescript
import 'dotenv/config';
import { defineConfig } from 'drizzle-kit';

export default defineConfig({
  out: './drizzle',
  schema: './src/db/schema.ts',
  dialect: 'postgresql',
  dbCredentials: {
    url: process.env.DATABASE_URL!,
  },
});
```

### What You Can Modify

✅ **ALLOWED**:
- `src/db/schema.ts` - Define/modify tables, columns, indexes

- `src/db/index.ts` - Database connection logic

❌ **DO NOT MODIFY**:
- `drizzle/*/migration.sql` - Generated SQL files
- `drizzle/*/snapshot.json` - Generated snapshots

## Migration Workflows

Drizzle v1 Beta offers **two primary workflows**:

### Workflow 1: `push` (Recommended for Development)

**Use when**: Rapid prototyping, local development, quick iterations

```bash
# 1. Modify schema in src/db/schema.ts
# 2. Push changes directly to database
cd backend && bun run drizzle-kit push
```

**What it does**:
1. Reads your current schema
2. Compares with database state
3. Generates ALTER statements
4. **Applies them immediately**
5. **Does NOT create migration files**

**Pros**:
- ✅ Fast iterations
- ✅ No migration file management
- ✅ Simple workflow

**Cons**:
- ❌ No migration history
- ❌ Can't review SQL before applying
- ❌ Not recommended for production

### Workflow 2: `generate` + `migrate` (Required for Production)

**Use when**: Production deployments, team collaboration, version control

```bash
# 1. Modify schema in src/db/schema.ts

# 2. Generate migration files
cd backend && bun run drizzle-kit generate

# 3. Review generated SQL in drizzle/[timestamp]_[name]/migration.sql

# 4. Apply migrations to database
cd backend && bun run drizzle-kit migrate
```

**What it does**:
1. **`generate`**: Creates SQL migration files by comparing schema changes
2. **`migrate`**: Applies unapplied migrations to the database
3. Tracks which migrations have been applied in `drizzle.__drizzle_migrations`

**Pros**:
- ✅ Full migration history
- ✅ Review SQL before applying
- ✅ Version control friendly
- ✅ Team-safe (handles conflicts)
- ✅ Production-ready

**Cons**:
- ❌ Slower than `push`
- ❌ More steps

## Making Schema Changes

### Location

All schema changes happen in: **`backend/src/db/schema.ts`**

### Current Schema Overview

```typescript
export const users = pgTable('users', { ... });
export const posts = pgTable('posts', { ... });
export const postFeedback = pgTable('post_feedback', { ... });
export const userUsage = pgTable('user_usage', { ... });
export const userActivity = pgTable('user_activity', { ... });
```

### Important UUID Note

**Always use** `gen_random_uuid()` for UUID defaults (PostgreSQL 13+ built-in):

```typescript
// ✅ CORRECT - Works on both PostgreSQL and Neon
id: uuid('id').primaryKey().default(sql`gen_random_uuid()`)

// ❌ WRONG - Requires uuid-ossp extension
id: uuid('id').primaryKey().default(sql`uuid_generate_v4()`)
```

## Local Development

### Prerequisites

- Docker running with `unslop-postgres` container
- `DATABASE_URL` set in `backend/.env`

```env
DATABASE_URL=postgresql://unslop:unslop_dev_password@localhost:5432/unslop
```

### Workflow

#### Option A: Using `push` (Quick iterations)

```bash
# 1. Start Docker
docker compose up -d

# 2. Modify schema
vim backend/src/db/schema.ts

# 3. Push changes
cd backend && bun run drizzle-kit push

# 4. Confirm changes when prompted
```

#### Option B: Using `generate` + `migrate` (Production-like)

```bash
# 1. Start Docker
docker compose up -d

# 2. Modify schema
vim backend/src/db/schema.ts

# 3. Generate migration
cd backend && bun run drizzle-kit generate

# 4. Review migration
cat drizzle/[timestamp]_[name]/migration.sql

# 5. Apply migration
cd backend && bun run drizzle-kit migrate
```

### Starting Fresh (Nuclear Option)

If you need a clean slate:

```bash
# 1. Drop everything (SQL allowed ONLY for this)
docker exec -i unslop-postgres psql -U unslop -d unslop -c "
DROP SCHEMA public CASCADE;
CREATE SCHEMA public;
GRANT ALL ON SCHEMA public TO unslop;
GRANT ALL ON SCHEMA public TO public;
DROP SCHEMA IF EXISTS drizzle CASCADE;
"

# 2. Delete old migrations
cd backend && rm -rf drizzle/2026*

# 3. Generate fresh migration
bun run drizzle-kit generate

# 4. Apply migration
bun run drizzle-kit migrate
```

## Production Deployment

### Neon (Our Production Database)

1. **Schema changes** happen in `src/db/schema.ts`
2. **Generate migration locally**:
   ```bash
   cd backend && bun run drizzle-kit generate
   ```
3. **Commit migration files** to Git
4. **Deploy** to production server
5. **Run migrations** during deployment:
   ```bash
   # In production environment
   cd backend && bun run drizzle-kit migrate
   ```

### CI/CD Integration

Add to your deployment pipeline:

```bash
# .github/workflows/deploy.yml or similar
- name: Run database migrations
  run: |
    cd backend
    bun run drizzle-kit migrate
  env:
    DATABASE_URL: ${{ secrets.NEON_DATABASE_URL }}
```

### Migration Safety

Drizzle automatically:
- ✅ Tracks applied migrations
- ✅ Only runs unapplied migrations
- ✅ Skips already-applied ones
- ✅ Runs migrations in order

## Common Operations

### Adding a New Table

```typescript
// backend/src/db/schema.ts
export const newTable = pgTable('new_table', {
  id: bigserial('id', { mode: 'number' }).primaryKey(),
  name: text('name').notNull(),
  createdAt: timestamp('created_at', { withTimezone: true }).notNull().defaultNow(),
});
```

**Then**:
```bash
cd backend && bun run drizzle-kit generate
cd backend && bun run drizzle-kit migrate
```

### Adding a Column

```typescript
// backend/src/db/schema.ts
export const users = pgTable('users', {
  // ... existing columns
  newColumn: text('new_column'), // ← Add this
});
```

**Then**:
```bash
cd backend && bun run drizzle-kit generate
cd backend && bun run drizzle-kit migrate
```

### Renaming a Column

Drizzle will **prompt you** during `generate`:

```bash
$ bun run drizzle-kit generate

? Detected renamed column:
  - old name: "old_name"
  - new name: "new_name"
  
  Is this a rename? (Y/n)
```

Answer `Y` to generate a `RENAME COLUMN` instead of `DROP` + `ADD`.

### Adding an Index

```typescript
// backend/src/db/schema.ts
export const posts = pgTable('posts', {
  postId: text('post_id').primaryKey(),
  authorId: text('author_id').notNull(),
  // ...
}, (table) => [
  index('idx_posts_author_id').on(table.authorId), // ← Existing
  index('idx_new_index').on(table.someColumn),     // ← Add this
]);
```

### Adding a Foreign Key

```typescript
export const newTable = pgTable('new_table', {
  userId: uuid('user_id').notNull().references(() => users.id),
});
```

### Dropping a Table

**⚠️ DANGER ZONE**

```typescript
// 1. Remove the table export from schema.ts
// export const oldTable = pgTable(...); ← DELETE THIS

// 2. Generate migration
cd backend && bun run drizzle-kit generate

// 3. Review the DROP TABLE statement
cat drizzle/[timestamp]_[name]/migration.sql

// 4. If safe, apply
cd backend && bun run drizzle-kit migrate
```

## Best Practices

### ✅ DO

1. **Always review generated SQL** before running `migrate`
2. **Test migrations locally first** (never test in production)
3. **Commit migration files** to version control
4. **Use `push` for local dev**, `generate + migrate` for production
5. **Keep schema.ts as single source of truth**
6. **Use descriptive table and column names**
7. **Add indexes for foreign keys**
8. **Use `gen_random_uuid()` for UUIDs** (not `uuid_generate_v4()`)

### ❌ DON'T

1. **Never edit generated migration.sql files**
2. **Never edit snapshot.json files**
3. **Don't skip migration files** (they must be sequential)
4. **Don't delete migration folders** (breaks history tracking)
5. **Don't use `push` in production**
6. **Don't manually run SQL migrations** (let Drizzle track them)
7. **Don't mix `push` and `generate` workflows** (pick one per environment)

### Migration Naming

Drizzle auto-generates names like: `20260204113836_cheerful_blink`

Format: `[TIMESTAMP]_[RANDOM_NAME]`

You can't customize the name, but the timestamp ensures ordering.

## Troubleshooting

### "Table already exists" Error

**Cause**: Database has tables that Drizzle doesn't know about.

**Solution**: Either:
1. Use `push` to sync (destroys migration history)
2. Mark migration as applied manually (only if desperately needed)

### "Column doesn't exist" After Migration

**Cause**: Migration wasn't applied or failed silently.

**Check**:
```bash
# View applied migrations
docker exec -i unslop-postgres psql -U unslop -d unslop -c "SELECT * FROM drizzle.__drizzle_migrations;"
```

### Type Errors After Schema Change

**Cause**: TypeScript is using old types.

**Solution**:
```bash
# Restart TypeScript server in your IDE
# Or run type check
cd backend && bun run tsc --noEmit
```

### Migration Conflicts

**Cause**: Two developers generated migrations at same time.

**Solution**:
1. Pull latest migrations from Git
2. Delete your local migration folder
3. Run `drizzle-kit generate` again (Drizzle will merge changes)
